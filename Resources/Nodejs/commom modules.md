
## المرحلة الأولى: الـ Module Wrapper (الغلاف الخفي لكودك)

بص يا وحش، لما بتكتب في ملفك `app.js` سطر واحد زي `const x = 5;`.. أنت متخيل إن الـ V8 Engine (المحرك اللي بيشغل الجافا سكريبت) بيشوف السطر ده لوحده؟ **الإجابة هي: مستحيل.**

لو نود نفذ كودك كدة "حاف"، فكل المتغيرات اللي هتعرفها هتبقى **Global**.. يعني لو عندك ملفين عرفت فيهم `const x` البرنامج هيضرب في وشك. عشان كدة نود اخترع "الخدعة الكبرى".

### 1. إيه هو الـ Wrapper كـ "Textual Transformation"؟

قبل ما نود يشغل كودك، بيتعامل مع الملف كأنه **String** (نص). بيقوم جايب "قالب" (Template) لـ Function جاهزة، ويقوم "حاشر" الكود بتاعك في نصها.

**الشكل اللي نود بيبنيه في الذاكرة (Memory) بيبقى كدة بالظبط:**

JavaScript

```
(function(exports, require, module, __filename, __dirname) {
// --------------------------------------------------
// هنا نود بيعمل Paste للكود اللي أنت كتبته في الملف
// --------------------------------------------------
});
```

دي بنسميها في الـ JavaScript الأساسية **IIFE** (Immediately Invoked Function Expression).

> **فلاش باك JS:** الـ IIFE هي فانكشن بتتعرف وتتنفذ في نفس اللحظة. فايدتها الأساسية في تاريخ الجافا سكريبت هي **Data Privacy**.. أي حاجة جواها محبوسة، ومحدش من بره يقدر يشوفها. ده السبب إنك لما بتعرف متغير في ملف، مبيشوفوش الملف التاني!

---

### 2. تشريح الـ Arguments الخمسة (حقن الأدوات - Dependency Injection)

نود مش بس بيلف كودك، ده "بيحقن" جواه 5 حاجات أنت بتستخدمهم وهم أصلاً مش موجودين في لغة الجافا سكريبت (لأن الـ JS كـ لغة مفيهاش `require` ولا `__dirname`).

#### أ. الـ `exports` و الـ `module` (الفرق الجوهري)

هنا بنحتاج نرجع لأساسيات الـ **Reference Types** في JS.

- نود بيكريت أوبجكت كبير اسمه `module`.
    
- جوه الأوبجكت ده فيه خاصية اسمها `exports` قيمتها أوبجكت فاضي `{}`.
    
- نود بيعمل متغير "طيار" اسمه `exports` بيشاور على نفس الأوبجكت الفاضي ده.
    

> **القاعدة:** `module` هو الـ "الحاوية" الحقيقية، و `exports` هو مجرد "اختصار" ليها.

#### ب. الـ `require` (مش مجرد فانكشن)

الـ `require` اللي بتستخدمها دي مش Global. نود بيفصل نسخة "Local" من الـ `require` لكل موديول. ليه؟ عشان لما تطلب ملف بمسار نسبي `./` الـ `require` تكون عارفة الملف "الأب" مكانه فين بالظبط على الهارد ديسك وتعرف تحسب المسار صح.

#### ج. الـ `__filename` و الـ `__dirname`

دول عبارة عن Strings نود بيحسبهم لكل ملف بناءً على مكانه في السيستم وبيديهم لك جاهزين عشان لو حبيت تقرأ ملفات أو تتعامل مع الـ File System.

---

### 3. سحر الـ `.call()` والـ `this` (ليه بنبعت `module.exports` مرتين؟)

ركز في السطر اللي نود بينفذه عشان يشغل الـ Wrapper ده، لأنه هو اللي بيجاوب على سؤالك:

JavaScript

```
compiledWrapper.call(module.exports, module.exports, require, module, filename, dirname);
```

#### يعني إيه `.call()` أصلاً؟ (فكرني يا سينيور)

في الجافا سكريبت، أي فانكشن ممكن تناديها بـ `()` عادي، بس لو ناديتها بـ `.call()`، أنت بتمسك "ريموت كنترول" الفانكشن دي.

- أول Argument في `.call()` بيحدد قيمة الـ **`this`** جوه الفانكشن.
    
- باقي الـ Arguments بتروح للـ parameters العادية.
    

**لما نود بعت `module.exports` في أول خانة:** خلى الـ `this` جوه ملفك تشاور على "شنطة الصادرات".

**ولما بعتها تاني في الخانة التانية:** راحت للـ parameter اللي اسمه `exports`.

**النتيجة المذهلة:** في أول سطر في ملفك، الـ `this` والـ `exports` والـ `module.exports` التلاتة بيشاوروا على **نفس الـ Object** في الميموري. عشان كدة أي واحد فيهم بتضيف فيه حاجة، بتسمع في الباقي.

---

### 4. الـ Execution Flow: إزاي الـ String بيبقى كود؟ (V8 Internals)

نود مبيستخدمش `eval()` لأنها "ثغرة أمنية" وبطيئة. نود بيستخدم موديول داخلي اسمه `vm` (Virtual Machine).

1. نود بياخد الـ Wrapper كـ **String**.
    
2. بيبعته للـ V8 Engine عن طريق `vm.runInThisContext()`.
    
3. الـ V8 بيعمل "Compile" للنص ده ويحوله لـ **Machine Code** (أصفار وحايد) ويفهم إن دي فانكشن.
    
4. نود يقوم منادي الـ Function الناتجة دي باستخدام `.call()`.
    

---

### 5. الـ Senior Interview Scenarios (إزاي تتسأل فيها؟)

#### السيناريو الأول: "The Global Leak"

**المحاور:** "يا خالد، لو أنا كتبت `x = 10;` من غير `const` ولا `let` جوه الملف، هل الـ Wrapper هيحرسني؟"

**إجابتك:** "لأ يا فندم. الـ Wrapper بيحبس الـ Local Variables بس. لكن لو كتبت متغير من غير تعريف (Undeclared)، الـ V8 هيدور عليه جوه الـ Wrapper مش هيلاقيه، هيطلع للـ Scope اللي فوقيه، واللي فوق الـ Wrapper في نود هو الـ **Global Object**. فالمتغير ده "هيهرب" ويبقى Global على مستوى الأبلكيشن كله. والحل هو استخدام `'use strict';` عشان نمنع الكارثة دي."

#### السيناريو الثاني: "The Reference Break"

**المحاور:** "ليه لو عملت `exports = { a: 1 }` الكود مبيصدرش حاجة؟"

**إجابتك:** "لأن `exports` مجرد Pointer بيشاور على `module.exports`. لما أنا سويته بـ Object جديد، أنا قطعت الـ Link ده. الـ `exports` دلوقتي بيشاور على مكان جديد، لكن نود لما بيخلص تنفيذ الـ Wrapper بيرجع الـ `module.exports` الأصلي اللي لسه فاضي! فالمطور اللي عملي require هيجيله `{}`."

## المرحلة التانية: رحلة الـ `require(X)` (خوارزمية البحث)

لما بتكتب `const x = require('something')`.. نود مبيخمنش، هو بيمشي على "كتالوج" خطوات صارم جداً. لو فشل في خطوة، بيتنقل للي بعدها بالترتيب ده:

### 1. الـ Core Modules (أصحاب الحصانة)

أول كلمة نود بيسألها لنفسه: **"هل 'something' ده موديول أساسي من بتوعي؟"** (زي `fs`, `path`, `http`, `crypto`).

- لو الإجابة "آه": بيحمله فوراً من الـ Binary بتاع نود (مكتوبين بـ C++ وجاهزين في الـ RAM).
    
- **سؤال سينيور:** "لو أنا عملت فولدر يدوي في `node_modules` وسميته `fs` وحطيت فيه كود، هل نود هيقرأه؟"
    
- **الإجابة:** **مستحيل.** نود بيدي أولوية قصوى للـ Core Modules. (إلا لو استخدمت الـ prefix الجديد `node:fs` وده اللي بننصح بيه دلوقتي عشان نمنع أي تداخل).
    

---

### 2. الـ File Modules (لما بتحدد المسار بنفسك)

لو الـ String اللي جوه الـ require بيبدأ بـ `/` (مسار مطلق) أو `./` (الفولدر الحالي) أو `../` (الفولدر اللي فوقك)، نود بيفهم إنك "بتشاور" على ملف بعينه.

**نود بيدور بالترتيب ده (الـ Extensions):**

1. بيدور على ملف اسمه `X` بالظبط.
    
2. لو ملقاش، بيدور على `X.js` (لأنه الأساس).
    
3. لو ملقاش، بيدور على `X.json` (بيحوله لـ Object أوتوماتيك بـ `JSON.parse`).
    
4. لو ملقاش، بيدور على `X.node` (ده ملف Binary متكود بـ C++).
    

> **فلاش باك JS:** نود هنا بيستخدم الـ `path` module داخلياً عشان يعمل "Normalize" للمسار، يعني لو أنت كاتب `//` زيادة أو حاطط نقط كتير، هو بينظف المسار الأول قبل ما يدور.

---

### 3. الـ Directory as a Module (الفولدر كأنه ملف)

لو `X` ده طلع "فولدر" مش ملف، نود مبييأسش، بيدخل جوه الفولدر ويدور بطريقتين:

- **الطريقة البريميم:** بيدور على ملف اسمه `package.json`. لو لقى جواه خاصية اسمها `"main"`، بيروح يشغل الملف اللي مكتوب قدامها.
    
    - مثال: لو مكتوب `"main": "./dist/app.js"`، نود هيشغل `app.js`.
        
- **الطريقة التقليدية:** لو ملقاش `package.json` (أو ملقاش خاصية main)، بيدور أوتوماتيك على ملف اسمه `index.js` أو `index.json`.
    
    - **الزتونة:** عشان كدة دايماً بنسمي الملف الأساسي في أي لايبراري `index.js`.. عشان نود يلقطه لوحده.
        

---

### 4. الـ `node_modules` (رحلة الصعود للجبل)

دي بقى الحتة اللي بتميز السينيور. لو أنت كتبت `require('express')` (لا هي Core ولا بتبدأ بـ `./`)، نود بيعمل عملية "تسلق" (Directory Climbing):

1. يبص في الفولدر الحالي: هل فيه `node_modules/express`؟
    
2. ملقاش؟ يطلع للفولدر اللي فوقه (Parent): هل فيه `../node_modules/express`؟
    
3. ويفضل طالع.. طالع.. لحد ما يوصل لآخر الهارد ديسك (الـ Root).
    

**ليه الحركة دي عبقرية؟**

عشان بتسمح لكل موديول يكون عنده الـ dependencies بتاعته الخاصة. لو موديول `A` محتاج نسخة قديمة من `lodash` وموديول `B` محتاج نسخة جديدة، نود هيعرف يوصل لكل واحدة في الـ `node_modules` القريبة منها.

## "السر الكبير": الـ Caching (ليه نود سريع وذكي؟)

ركز يا خالد عشان دي بتيجي في الإنترفيوهات كتير. **"لو عملت require لملف 10 مرات، الكود اللي جواه بيتنفذ كام مرة؟"**

**الإجابة:** **مرة واحدة بس.**

نود عنده أوبجكت سري في الميموري اسمه `require.cache`.

- أول مرة بتطلب الملف: نود بيقرأه من الهارد ديسك (عملية بطيئة)، يلفه بالـ Wrapper، ينفذه، وياخد الـ `module.exports` بتاعته يخزنها في الـ Cache.
    
- تاني مرة: نود بيبص في الـ Cache (عملية سريعة جداً في الـ RAM)، يلاقيه موجود، يديك النسخة اللي عنده فوراً **من غير ما يعيد تنفيذ الكود**.
    

> **تنبيه سينيور:** لو أنت حاطط `console.log` بره الـ functions في ملف، الـ log ده هيظهر **مرة واحدة بس** مهما عملت require للملف ده في أماكن تانية.

### 5. الـ Senior Interview Scenarios (مطبخ الشغل)

#### السيناريو الأول: "The Singleton Pattern"

**المحاور:** "يا خالد، إزاي أضمن إن الأوبجكت بتاع الـ Database Connection يكون واحد بس في الأبلكيشن كله؟"

**إجابتك:** "ببساطة يا فندم، هعمل ملف `db.js` وأعمل فيه `new Connection()` وأطلعه بـ `module.exports`. بسبب الـ **Caching** في نود، أي حد في الأبلكيشن هيعمل `require('./db')` هيجيله نفس الـ Instance بالظبط، لأن نود نفذ الكود وكريت الأوبجكت مرة واحدة بس وشاله في الـ Cache."

#### السيناريو الثاني: "The Case-Sensitivity Nightmare"

**المحاور:** "لو أنا في لينكس وطلبت ملف اسمه `User.js` وأنا كاتبه في الكود `require('./user')` (u small)، إيه اللي هيحصل؟"

**إجابتك:** "نود هيطلع Error `MODULE_NOT_FOUND`. لأن نود في الـ Resolution بيعتمد على الـ File System بتاع نظام التشغيل. لينكس (Case-sensitive) هيشوفهم ملفين مختلفين، عكس ويندوز اللي ممكن يعديها. ودي غلطة جونيورز كتير بتقع فيها لما بيرفعوا كودهم على السيرفر (اللي غالباً لينكس)."

## المرحلة التالتة: تشريح الـ `module` Object (هوية الموديول)

لما تكتب `console.log(module)` جوه أي ملف، هيطلعلك أوبجكت فيه شوية "Properties" مرعبة. تعال نفصصهم "حتة حتة" بالتفصيل الممل:

### 1. الـ `module.id` والـ `module.filename` (بطاقة الرقم القومي)

- **`module.filename`:** ده هو المسار المطلق (Absolute Path) للملف على الهارد ديسك. يعني بيبدأ من الـ `C:/` أو الـ `/home/` لحد اسم الملف.
    
- **`module.id`:** غالباً بيبقى هو هو الـ `filename`.
    
- **الزتونة للسينيور:** ليه نود مهتم بالـ ID؟ عشان الـ **Caching**! نود بيستخدم الـ ID ده كـ "Key" في أوبجكت الـ Cache. لو حاولت تعمل require لملفين ليهم نفس الـ ID، نود هيعرف إنهم ملف واحد وهيحمل النسخة اللي في الميموري فوراً.
    
- **تريك الانترفيو:** "إيه هو الـ ID بتاع أول ملف بنشغله (Entry Point)؟"
    
    - **الإجابة:** الـ ID بتاعه بيبقى عبارة عن نقطة `.` بس كاختصار، أما باقي الملفات اللي بيحصلها require فبتاخد الـ Full Path.
        

---

### 2. الـ `module.loaded` (عين نود اللي مابتنامش)

ده عبارة عن `Boolean` (يا `true` يا `false`).

- بص يا خالد، نود وهو بيقرأ ملفك، الـ `loaded` دي بتبقى `false`. أول ما يوصل لآخر سطر في الـ Wrapper ويخلص تنفيذ، نود بيقلبها `true`.
    
- **ليه دي مهمة؟** فاكر الـ **Circular Dependency** اللي شرحناها في المرحلة التانية؟ نود بيبص على الـ `loaded` دي. لو ملف `A` طلب `B` و `B` رجع طلب `A`.. نود بيشوف إن `A.loaded` لسه بـ `false`.. فيقوم مديله "النسخة اللي لحق يخلصها" عشان يفك الاشتباك.
    

---

### 3. الـ `module.children` والـ `module.parent` (شجرة العيلة)

دي بقى اللي بتخليك تتخيل الأبلكيشن كأنه "شجرة" (Tree) مش مجرد ملفات سايحة.

- **`module.children`:** دي Array فيها كل الملفات اللي أنت عملتلها `require` جوه ملفك الحالي.
    
- **`module.parent`:** ده كان الأوبجكت اللي بيشاور على "مين اللي ناداني؟".
    
    - **تحذير سينيور:** الـ `module.parent` دلوقتي بقت **Deprecated** (قديمة ومبقتش مفضلة) في نود. ليه؟ لأن الملف ممكن يحصله require من أكتر من مكان، فكلمة "أب واحد" دي مبقتش دقيقة. البديل بتاعها هو الـ `require.main` اللي هنشرحه كمان شوية.
        

---

### 4. الـ `module.paths` (خريطة البحث)

دي Array من الـ Strings. فاكر لما قلنا نود "بيصعد الجبل" عشان يدور في الـ `node_modules`؟ الـ `module.paths` هي دي "الخريطة" اللي نود بيمشي عليها.

- بيجيب المسار الحالي ويضيف عليه `/node_modules`.
    
- يطلع للي فوقه ويضيف `/node_modules`.
    
- السينيور الشاطر يقدر يبص على الـ Array دي لو عنده مشكلة إن نود بيقرأ نسخة غلط من مكتبة معينة، عشان يعرف نود "ناوي" يدور فين بالظبط.

## "الضربة القاضية": الـ `require.main === module` (التريك اللي بتموت أي جونيور)

دي أهم حتة في المرحلة دي يا خالد. تخيل إنك عامل ملف اسمه `utility.js` فيه شوية functions، وعايزه يعمل حاجتين:

1. لو حد عمله `require`.. يديله الـ functions بس.
    
2. لو أنت شغلته بنفسك من الترمينال (`node utility.js`).. ينفذ شوية كود تجريبي (Test code) أو يطبع حاجة.
    

**إزاي الملف يعرف هو شغال "لوحده" ولا "حد مناديه"؟**

JavaScript

```
// utility.js

function calculate(x) { return x * 2; }

if (require.main === module) {
  // الكود اللي هنا مش هيشتغل إلا لو كتبت node utility.js
  console.log('أنا شغال لوحدي يا خالد! النتيجة اهي:', calculate(10));
} else {
  // الكود اللي هنا هيشتغل لو حد عمل require('./utility')
  console.log('حد ناداني كـ موديول..');
}

module.exports = { calculate };
```

**شرح العمق:**

- **`require.main`:** ده pointer دايماً بيشاور على الـ "Entry Point" (أول ملف ناديت عليه في الترمينال).
    
- **`module`:** هو الأوبجكت بتاع الملف الحالي.
    
- لما تسويهم ببعض، أنت بتسأل: "هل أنا هو الملف الأساسي اللي بدأنا بيه الحفلة دي؟". لو آه، نفذ كود الـ CLI.

## المرحلة الرابعة: خناقة الـ `exports` والـ `module.exports`

عشان تفهم دي، لازم نرجع لثانية واحدة لدرس "الرابر" (Wrapper) اللي شرحناه في الأول. فاكر السطر ده؟

`function (exports, require, module, __filename, __dirname) { ... }`

### 1. الحقيقة الصادمة (The Memory Link)

في بداية تشغيل أي موديول، نود بيكريت أوبجكت في الميموري (خلينا نسميه **Object A**). وبعدين بيعمل حاجتين:

1. بيخلي `module.exports` تشاور على **Object A**.
    
2. بيبعت الـ Reference بتاع **Object A** ده للـ Argument اللي اسمه `exports`.
    

**النتيجة:** في اللحظة دي، `module.exports === exports` هي `true`. الاتنين عبارة عن "مؤشرين" (Pointers) لنفس المكان في الميموري.

---

### 2. ليه نود عمل "الدوشة" دي؟ (The Helper Logic)

المصممين بتوع نود لقوا إن المطورين بيكتبوا `module.exports` كتير قوي، فقالوا: "يا جماعة، إحنا هنبعتلكم متغير صغير اسمه `exports` يكون مجرد **Shorthand** (اختصار) للـ `module.exports` الأصلية، عشان توفروا مجهود".

> **فلاش باك JS (Reference vs Value):** > في الجافا سكريبت، الأوبجكتس بتتنقل بالـ **Reference**. يعني لو عندك `let a = { name: 'Khaled' };` وقولت `let b = a;`.. الـ `b` مش نسخة تانية، هي مجرد "اسم تاني" لنفس الأوبجكت. لو غيرت `b.name` الـ `a.name` هتتغير فوراً.

---

### 3. متى تحدث الكارثة؟ (The Reassignment Trap)

هنا بقى يا خالد الفرق بين السينيور والجونيور.

#### السيناريو الأول (الصح ✅):

JavaScript

```
exports.add = (a, b) => a + b;
exports.greet = () => "Hello";
```

هنا أنت بتعدل "جوه" الأوبجكت اللي `exports` بيشاور عليه. وبما إن `module.exports` بيشاور على نفس الأوبجكت، فالتعديلات دي بتسمع فيه، ونود في الآخر بيعمل `return module.exports` فالدنيا بتبقى تمام.

#### السيناريو الثاني (الكارثة ❌):

JavaScript

```
exports = {
  add: (a, b) => a + b,
  greet: () => "Hello"
};
```

**إيه اللي حصل هنا؟** أنت هنا عملت **Reassignment**. أنت قولت للـ Pointer اللي اسمه `exports`: "سيب الأوبجكت القديم اللي كنت بتشاور عليه، وشاور على الأوبجكت الجديد ده".

**المشكلة:** الـ `module.exports` لسه بيشاور على الأوبجكت القديم (الفاضي)! نود ملوش دعوة بكلمة `exports` بتاعتك دي، هو في الآخر بيرجع للي عمل `require` الـ `module.exports`.

---

### 4. القاعدة الذهبية (The Golden Rule)

لو عايز تغير الـ Export بالكامل (يعني تخليه `Function` بدل `Object` أو تخليه `Class`)، لازم تستخدم **`module.exports`** حصرياً.

**مثال السينيور:**

JavaScript

```
// لو عايز تصدر فانكشن واحدة بس هي اللي تشتغل لما حد ينادي الموديول
module.exports = function() {
  console.log("I am the only one!");
};

// لو جربت تعمل exports = function... مش هتشتغل للسبب اللي قلناه فوق
```

### 5. الـ Senior Interview Scenarios (أسئلة ليفل الوحش)

#### السيناريو الأول: "The Mixed Exports"

**المحاور:** "يا خالد، لو كتبت الكود ده، إيه اللي هيحصل؟"

JavaScript

```
exports.name = "Khaled";
module.exports = { age: 25 };
```

**إجابتك:** "الموديول هيصدر `{ age: 25 }` بس. لأن سطر `module.exports = ...` عمل Overwrite للأوبجكت القديم بالكامل وكريت أوبجكت جديد، فكل اللي ضفته عن طريق `exports` في الأول ضاع، لأن `exports` لسه بيشاور على الأوبجكت القديم اللي نود رماه في الزبالة (Garbage Collected)."

#### السيناريو الثاني: "Exports as a Function"

**المحاور:** "هل ينفع أخلي الموديول يرجع Function وفي نفس الوقت يكون ليها Properties؟"

**إجابتك:** "آه طبعاً، بس لازم أستخدم `module.exports`. هخلي `module.exports` بتساوي الفانكشن، وبعدين أضيف عليها properties عادي، لأن الفانكشن في JS هي في الأصل Object."

JavaScript

```
module.exports = () => "Main Function";
module.exports.subTask = () => "Sub Task";
```

---

### ملخص الـ CommonJS اللي هتقفل بيه أي إنترفيو:

1. **الـ Wrapper:** هو اللي بيعزل الكود وبيدينا الأدوات (`require`, `module`, etc).
    
2. **الـ Resolution:** نود بيدور بالترتيب (Core -> File -> node_modules).
    
3. **الـ Caching:** الملف بيتنفذ مرة واحدة بس وبيتحفظ في الميموري بالمسار الكامل بتاعه.
    
4. **الـ `this`:** هي هي `module.exports` في الـ top-level بسبب الـ `.call()`.
    
5. **الـ `exports`:** مجرد اختصار (Reference)، إياك تعمل له Reassign.

## الجزء الأول: أسئلة الـ Senior Interview (Scenario-based)

الأسئلة دي مش "ما هو الـ Wrapper"، دي أسئلة مواقف برمجية حقيقية:

### السؤال الأول: "لغز الـ `this` والـ Arrow Function"

**المحاور:** "يا خالد، لو أنا عرفت موديول بستخدم فيه Arrow Function في الـ top-level، وحاولت أستخدم `this.x = 10` عشان أعمل export، هل الكود ده هيشتغل؟ وليه؟"

- **الإجابة المتوقعة:** "آه هيشتغل. لأن الـ Arrow function في الـ top-level بتاع الموديول بتاخد الـ `this` من الـ Lexical Scope بتاعها، وهو هنا الـ Wrapper function. وبما إن نود نادى الـ Wrapper بـ `.call(module.exports)`، فالـ `this` هتكون هي الـ `module.exports` فعلاً."
    

### السؤال الثاني: "كارثة الـ `exports` الضائع"

**المحاور:** "بص على الكود ده وقولي الـ `require` هترجع إيه؟"

JavaScript

```
// user.js
exports.name = "Khaled";
module.exports = { job: "Senior Dev" };
```

- **الإجابة المتوقعة:** "هترجع `{ job: "Senior Dev" }` بس. لأن السطر التاني عمل Overwrite (إعادة تعيين) لـ `module.exports` بأوبجكت جديد خالص، والـ `exports` القديم اللي ضفنا فيه الـ `name` فضل بيشاور على الأوبجكت القديم اللي نود أهمله."
    

### السؤال الثالث: "فخ الـ `require.cache`"

**المحاور:** "عندي ملف `config.json`. غيرت فيه يدوي على الهارد ديسك والأبلكيشن شغال. لما عملت `require` تانية لنفس الملف، نود قرأ الداتا القديمة. إزاي أجبر نود يقرأ الداتا الجديدة من غير ما أرستر السيرفر؟"

- **الإجابة المتوقعة:** "لازم أمسح الـ Cache يدوياً. هجيب الـ Absolute path بتاع الملف بـ `require.resolve('./config.json')` وبعدين أعمل `delete require.cache[path]`. كدة نود المرة الجاية هيضطر يروح للهارد ديسك ويقرأه من جديد."
